# Cloudflare R2 uploads — DogShift

## Why R2

S3-compatible storage via AWS SDK, no egress fees, GDPR-friendly EU region. Used for :
- Sitter avatars
- Verification documents (identity, OPAn certificate, pension photos)
- Signed contract PDFs

## Two-phase pattern (NEVER bypass)

### Phase 1 — Presign (server)

User uploads small file metadata to your route. Server :
1. Auths the user
2. Generates a unique key with the **sitter's prefix** : `<sitterId>/<scope>/<uuid>-<filename>`
3. Calls R2 to get a temporary PUT URL valid 60-120 s
4. Returns `{ url, key }` to client

```ts
const expectedPrefix = `${sitterId}/avatar/`;
const key = `${expectedPrefix}${randomUUID()}-${sanitizeFilename(filename)}`;
const url = await presignPutUrl(key, contentType, 60);
return NextResponse.json({ url, key });
```

### Phase 2 — Client upload

```ts
await fetch(presignedUrl, {
  method: "PUT",
  headers: { "Content-Type": file.type },
  body: file,
});
```

Direct browser → R2. No bytes through your server.

### Phase 3 — Commit (server)

Client posts the `key` back. Server :
1. Auths the user **again** (presign auth may have expired)
2. **CRITICAL prefix check** : `key.startsWith(expectedPrefix)` — without this, a sitter could presign their own avatar then commit a different sitter's verification doc
3. Optional : HEAD R2 to confirm the file actually landed
4. Updates DB with the key / public URL

```ts
const expectedPrefix = `${sitterId}/avatar/`;
if (!isSitterAvatarR2Key(key) || !key.startsWith(expectedPrefix)) {
  return NextResponse.json({ ok: false, error: "INVALID_KEY" }, { status: 400 });
}
// safe to commit
await prisma.sitterProfile.update({ where: { id }, data: { avatarUrl: publicUrlFor(key) } });
```

## The prefix check is the ONLY security gate

R2 itself is wide-open on the bucket — anyone with a presigned URL can PUT to that key. The presigned URL is scoped to the key you sign, but if you let a malicious user commit ANY key (not just theirs), you've handed them write access to anyone's avatar / verif doc.

**Always `key.startsWith(expectedPrefix)` on commit.** Always.

## Contract PDFs are different

Contract PDFs are uploaded **server-side** after the sitter signs — no presign, no two-phase. The signed PDF is generated by the server then PUT to R2 in one shot. Path : `app/api/contract/sign/[token]/route.ts`.

## Env vars

```bash
CLOUDFLARE_R2_ACCOUNT_ID=
CLOUDFLARE_R2_ACCESS_KEY_ID=
CLOUDFLARE_R2_SECRET_ACCESS_KEY=
CLOUDFLARE_R2_BUCKET=
```

R2 endpoint pattern : `https://<account>.r2.cloudflarestorage.com/<bucket>/<key>`

When adding a new upload, copy the closest existing pair as starting point.

## What NOT to do

- ❌ Skip the prefix check on commit → cross-sitter overwrite vulnerability
- ❌ Use predictable keys like `${sitterId}.jpg` → enumeration leak. Always include a UUID.
- ❌ Stream the file through your server (slow, eats Vercel function memory)
- ❌ Trust the client's reported file size / type without server-side validation post-upload
- ❌ Use the same presign for multiple files (one URL, one PUT)
- ❌ Embed the R2 secret in any client-side code (only ACCOUNT_ID is safe, but even that should stay server-side)

## Public URLs

Public URLs go through a Cloudflare worker that adds caching + CORS :

```
https://r2.dogshift.ch/<key>
```

NOT the raw `<account>.r2.cloudflarestorage.com` URL (slow + leaks account).

For avatars served with auth tokens (sitter previews), use `/api/media/sitter-avatar/[token]` which generates short-lived signed URLs.

## Debugging "upload returns 200 but file not in R2"

- Check the R2 bucket directly via Cloudflare dashboard → R2 → Buckets → search by key prefix
- Check CORS config on the bucket (must allow `PUT` from `https://www.dogshift.ch`)
- Check the presigned URL expired (default 60 s — slow uploads timeout)
- Check the client sent the same `Content-Type` it presigned for (mismatch = 403 from R2)
