# Ship a PR — DogShift workflow

## When to invoke

- User says "ship", "push", "fais une PR", or similar
- A non-trivial code change is complete and tested locally
- A bug fix needs to land in prod (always go through a PR — `main` is branch-protected)

**Don't ship**:
- Brain notes (`brain/` is gitignored)
- WIP code
- Without a clear commit message in mind

## Branch naming

Strict convention — readable `gh pr list`:

| Prefix | Use for |
|---|---|
| `feat/<slug>` | New user-facing feature |
| `fix/<slug>` | Bug fix (especially prod regressions) |
| `chore/<slug>` | Tooling, deps, brain scripts, internal cleanup |
| `docs/<slug>` | Docs only (no code change) |
| `perf/<slug>` | Performance optimization |
| `hotfix/<slug>` | Direct prod fix when CI is queued |
| `ship/<ts>-<slug>` | Only when generated by `npm run ship` |

Never: `bugfix/`, `feature/`, arbitrary names.

## Workflow

### Option A — `npm run ship` (happy path)

```bash
npm run ship -- "fix(host): commit message subject"
```

Does branch + commit + push + PR + auto-merge in one shot. Use when:
- You're confident the change is small/clean
- The commit message follows Conventional Commits
- CI is expected to pass

### Option B — manual (recommended for non-trivial PRs)

```bash
git checkout -b <prefix>/<slug>
git add <specific files — NOT git add -A>
git commit -m "$(cat <<'EOF'
<type>(<scope>): subject (max 70 chars)

<body — explain WHY, not WHAT. Include:
- Context that's not in the diff (a Sentry alert, user report, …)
- Hypotheses considered + rejected
- Anti-pattern reminders for future-Claude>

Co-Authored-By: Claude Opus 4.6 <noreply@anthropic.com>
EOF
)"
git push -u origin <branch>
gh pr create --title "..." --body "..."
gh pr merge <pr-number> --auto --squash
```

## Commit message rules

- Subject = imperative, max 70 chars, lowercase after type, no trailing period
- Type ∈ `feat | fix | chore | docs | refactor | perf | test`
- Body: focus on **why**, not **what** (the diff shows the what)
- Always end with `Co-Authored-By: Claude Opus 4.6 <noreply@anthropic.com>`
- Use a HEREDOC (`-m "$(cat <<'EOF' ... EOF)"`) so newlines and quotes render correctly

## File staging — be specific

```bash
# ✅ Good — explicit
git add app/api/cron/foo.ts vercel.json tests/integrations/foo.test.ts

# ❌ Bad — drags in macOS Finder duplicates ("foo 2.tsx"), secrets, etc.
git add -A
git add .
```

The dogshift repo currently has macOS Finder duplicates lurking (`* 2.tsx`, `* 2.ts`) — never commit those.

## CI gating

Auto-merge is enabled — GitHub merges the PR when:
- `quality` job passes (lint, typecheck, unit tests, Next build)
- `e2e` job passes (Playwright smoke against Vercel preview)

If a job is `IN_PROGRESS`, just wait — don't poll aggressively. Use:

```bash
gh pr view <num> --json state,mergedAt,statusCheckRollup
```

Once. If still in progress, wait 60-120s and check again.

## When to skip hooks (`--no-verify`)

**Almost never.** Husky's `lint-staged` blocks commits on lint errors. If you hit one:

1. Fix the lint issue (preferred — usually `<a href>` → `<Link>` or unused imports)
2. Pre-existing lint debt in a file you're editing → fix it as a drive-by
3. Last resort, only if the user explicitly authorizes: `git commit --no-verify`

## Post-merge verification

For routes / crons:

```bash
# Wait for Vercel deploy
sleep 180

# Smoke test (replace with the actual route)
CRON_SECRET=$(grep ^CRON_SECRET= .env.local | cut -d= -f2- | tr -d '"' | tr -d "'")
curl -H "Authorization: Bearer $CRON_SECRET" "https://www.dogshift.ch/api/cron/<route>?force=1"
```

For UI changes: ask the user to reload the page in their browser.

## Bug-fix discipline

Every bug fix PR **must**:
1. Add at least one regression test in `tests/<topic>/`
2. Add or update a fiche in `docs/bugs/<symptom-slug>.md` with the `## 🤖 Automated detection` JSON block
3. Call `reportApiError()` in any new `catch` block (Sentry alert convention)

The nightly cron at 02:07 UTC reads every fiche and runs the detection block — no manual wiring needed.

## Anti-patterns

- ❌ `git push --force` to `main` — branch is protected anyway, but never attempt
- ❌ `git commit --amend` after the commit was pushed — creates a new commit instead
- ❌ Merging a PR with red CI by passing `--admin` — investigate first; only `gh pr merge --admin` after explicit user approval AND understanding why CI failed
- ❌ Combining unrelated changes in one PR — split them
- ❌ Skipping the regression test — locks in the bug recurrence
